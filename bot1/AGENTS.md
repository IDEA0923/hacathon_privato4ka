# AGENTS.md — Hackathon Project

## 1. Назначение проекта

Этот репозиторий предназначен для проекта на хакатон. Точная тема, критерии оценки и стек пока неизвестны; после получения задания этот файл нужно обновить под конкретные требования.

Главная цель: быстро собрать **работающий, демонстрируемый MVP**, который решает одну понятную проблему и стабильно запускается на защите.

- Название проекта: `[добавить позже]`
- Проблема, которую решает проект: `[добавить позже]`
- Целевая аудитория: `[добавить позже]`
- Формат демонстрации: `[сайт / приложение / бот / ноутбук / API / другое]`
- Дедлайн и ограничения: `[добавить позже]`
- Критерии жюри: `[добавить позже]`
- Выбранный стек: `[добавить позже]`

---

## 2. Главные приоритеты

Работай в таком порядке:

1. Понять задачу и определить минимальный MVP.
2. Сделать основной пользовательский сценарий от начала до конца.
3. Добиться стабильного запуска и понятной демонстрации.
4. Добавить тесты и обработку основных ошибок.
5. Проверить безопасность и отсутствие секретов в репозитории.
6. Улучшать дизайн, дополнительные функции и оптимизацию только после работающего MVP.

Не усложняй архитектуру ради красоты, если более простое решение достаточно для хакатона.

---

## 3. Правила работы Codex

Перед началом любой задачи:

1. Прочитай этот файл, `README.md`, `PLANS.md`, если он существует, и относящиеся к задаче файлы.
2. Кратко опиши, что уже есть в проекте и что требуется изменить.
3. Не переписывай рабочие части проекта без необходимости.
4. Делай изменения небольшими, проверяемыми шагами.
5. После изменений перечисли затронутые файлы, что изменено и как это проверить.

Если задача затрагивает несколько модулей, добавляет новую функцию, меняет архитектуру, подключает внешний API или требует нескольких этапов, сначала создай или обнови `PLANS.md` с кратким планом реализации.

Если пользователь просит только анализ или ревью, **не меняй код**, пока пользователь прямо не попросит внести исправления.

---

## 4. Предпочтительный стек до уточнения темы

По умолчанию предпочитай:

- основной язык: **Python 3.11+**;
- управление зависимостями: `venv` и `requirements.txt` либо `pyproject.toml`, если он уже используется;
- тестирование: `pytest`;
- переменные окружения и секреты: `.env`, но файл `.env` никогда не коммитить;
- форматирование и читаемость: понятные имена, небольшие функции, type hints для важных функций;
- документация запуска: `README.md`.

Не меняй язык, фреймворк или способ запуска проекта без причины. Если условие хакатона требует другой стек, обнови этот раздел.

---

## 5. MVP и ограничения объёма

До реализации новых функций сформулируй:

- какую проблему решает MVP;
- один главный пользовательский сценарий;
- какие функции обязательны для демонстрации;
- какие идеи откладываются как дополнительные;
- как проверить, что MVP готов.

Во время хакатона предпочитай одну полностью рабочую функцию нескольким недоделанным.

### Базовый главный сценарий

`[Пользователь] → [вводит/загружает данные] → [система обрабатывает] → [пользователь получает полезный результат]`

Заменить этот сценарий после выбора идеи.

---

## 6. Правила написания кода

- Пиши код так, чтобы его мог понять начинающий Python-разработчик.
- Не добавляй сложные паттерны, абстракции и дополнительные слои без реальной пользы.
- Не дублируй логику: повторяющийся код выноси в функцию или модуль, если это действительно упрощает проект.
- Добавляй короткие комментарии только там, где решение неочевидно.
- Не оставляй большие закомментированные фрагменты старого кода.
- Не создавай лишние файлы и зависимости.
- Сохраняй существующий стиль проекта, если он уже сформировался.
- При изменении поведения обновляй документацию и тесты, когда это уместно.

---

## 7. Зависимости и внешние сервисы

- Перед добавлением новой библиотеки объясни, зачем она нужна и почему стандартной библиотеки недостаточно.
- Предпочитай популярные, поддерживаемые и простые зависимости.
- Не подключай платные или требующие регистрации сервисы без явной необходимости.
- Не меняй версии библиотек массово без причины.
- Все необходимые зависимости должны быть зафиксированы в файле зависимостей.

Если проект зависит от внешнего API, предусмотрите режим понятной ошибки или демо-сценарий на случай отсутствия ключа или сбоя API.

---

## 8. ИИ, модели и API

Если проект связан с искусственным интеллектом:

- сначала реализуй самый простой рабочий сценарий вызова модели;
- не добавляй сложную агентную архитектуру, RAG, векторную базу или несколько моделей, пока это не требуется задачей;
- системные промпты и шаблоны запросов храни отдельно от основной логики, если они становятся большими;
- проверяй ответы модели на пустые, ошибочные и слишком длинные результаты;
- не считай вывод модели гарантированно правильным;
- при необходимости показывай пользователю, что ответ создан ИИ;
- для демонстрации подготовь несколько заранее проверенных примеров ввода.

Для OpenAI API используй актуальную официальную документацию через Skill `openai-docs`, если он доступен.

---

## 9. Безопасность и данные

Обязательные правила:

- Никогда не записывай настоящие API-ключи, пароли, токены или приватные данные в код, тесты, README, логи или Git.
- Используй переменные окружения для секретов.
- Добавь `.env` в `.gitignore`; при необходимости создай безопасный `.env.example` только с названиями переменных и фиктивными значениями.
- Не отправляй пользовательские данные во внешние сервисы без необходимости для работы функции.
- Не выполняй пользовательский ввод как код.
- Не используй `eval()` или `exec()` для обработки данных пользователя или содержимого файлов.
- Валидируй входные данные и обрабатывай ошибки внешних API.
- Не выводи секреты в консоль или интерфейс.

Перед финальной сдачей проведи security-проверку проекта. При наличии Skill используй `security-best-practices` сначала в режиме отчёта, без автоматического переписывания всего проекта.

## 10. Тестирование и проверка результата

После каждого значимого изменения:

1. Запусти проект или относящуюся к задаче часть.
2. Запусти существующие тесты.
3. Добавь или обнови тесты для новой основной логики, если это возможно за разумное время.
4. Проверь обработку хотя бы одного неверного или пустого ввода.
5. Сообщи пользователю, какие проверки реально были выполнены, а какие не удалось выполнить.

### Команды по умолчанию для Python-проекта

После создания проекта заменить этот блок точными командами.

- Создание виртуального окружения: `python -m venv .venv`
- Активация в Windows PowerShell: `.\.venv\Scripts\Activate.ps1`
- Установка зависимостей: `python -m pip install -r requirements.txt`
- Запуск тестов: `python -m pytest`
- Запуск приложения: `[добавить после выбора типа проекта]`

Не утверждай, что тесты прошли, если ты их не запускал или не видел результат.

## 11. Терминал и объяснение команд

Когда предлагаешь пользователю выполнить команду в терминале, всегда кратко объясняй:

- что команда делает;
- что означают важные части команды и аргументы;
- зачем этот шаг нужен именно сейчас;
- какой результат ожидается после выполнения.

Не давай «голые» команды без объяснения, особенно для установки пакетов, Git, Linux, Docker, запуска серверов и работы с файлами.

---

## 12. Git и сохранность проекта

Перед коммитом или отправкой проекта:

- проверь `git status`;
- проверь изменения через `git diff` и, если есть staged-файлы, через `git diff --staged`;
- убедись, что в репозиторий не попали секреты и временные файлы;
- коммить только понятные изменения, связанные с текущей задачей.

Не добавлять в Git без отдельной причины:

- `.env` и любые реальные ключи/токены;
- `.venv/`;
- `__pycache__/`, `.pytest_cache/` и временные файлы;
- локальные отчёты и логи с пользовательскими данными;
- большие датасеты и модели, если это не требуется условиями хакатона.

## 13. Документация и демонстрация

К моменту сдачи проект должен иметь понятный `README.md`, где указано:

- что делает проект;
- какую проблему решает;
- используемый стек;
- как установить зависимости;
- как указать переменные окружения без раскрытия ключей;
- как запустить проект;
- как воспроизвести главный демонстрационный сценарий;
- какие ограничения у текущей версии.

Для защиты подготовь:

- стабильный сценарий демонстрации;
- тестовые входные данные;
- запасной вариант: скриншоты или короткую запись работы, если живой запуск подведёт;
- краткое объяснение, где именно используется ИИ и какую пользу он даёт.

## 14. Работа со Skills в Codex

Используй Skills только тогда, когда они реально подходят к текущей задаче.

Ориентировочно:

- `define-goal` — в начале, чтобы определить MVP и критерии готовности;
- `openai-docs` — при работе с OpenAI API;
- `jupyter-notebook` — для экспериментов с данными, ML или сравнения AI-вариантов;
- `playwright` — если появляется веб-интерфейс, который нужно проверить в браузере;
- `security-best-practices` — перед сдачей или после добавления API/авторизации/обработки данных;
- Skill деплоя — только после выбора конкретной платформы публикации.

Не подключай множество Skills «на всякий случай». Сначала нужна задача, затем подходящий инструмент.

---

## 15. Definition of Done — когда задача считается завершённой

Изменение считается готовым, только если:

- реализована именно запрошенная функция, без лишнего расширения объёма;
- основной сценарий работает;
- проект запускается или явно описано, почему запуск не был выполнен;
- тесты запущены либо честно указано, почему их нет или не удалось запустить;
- нет настоящих секретов и опасных данных в изменённых файлах;
- README или инструкции обновлены, если изменился способ запуска или использования;
- результат можно понятно показать на демонстрации.

## 16. Что обновить после получения темы хакатона

Сразу после объявления задания обновить этот файл:

1. Указать проблему, аудиторию и точный MVP.
2. Записать критерии оценки и дедлайн.
3. Зафиксировать стек, структуру проекта и команды запуска.
4. Добавить главный пользовательский сценарий.
5. Выбрать необходимые Skills и удалить неактуальные рекомендации.
6. Уточнить тесты, правила деплоя и формат демонстрации.
7. Создать `PLANS.md` с первым реальным планом реализации.
