# AI-рекрутер олимпиад — Frontend

Фронтенд сервиса-агрегатора российских олимпиад, конкурсов и научных конференций для школьников. По профилю ученика (предметы, классы, регион, уровень) интерфейс показывает персональный список олимпиад с дедлайнами, AI-обоснованиями и календарём.

Это документация **только по фронтенду**: какие страницы есть, какие модули за что отвечают, как они связаны и какие данные между ними ходят. Бэкенд (C++) описан отдельно, фронт умеет работать и без него — через mock-режим.

## Стек

- **HTML + CSS + ванильный JavaScript** (ES-модули, без сборщика и фреймворков).
- **Nginx** раздаёт статику и проксирует `/api/*` на бэкенд.
- **Docker / docker-compose** для запуска.
- **LocalStorage** как клиентское хранилище (`user_id`, профиль, план, флаг mock).

Никаких npm-зависимостей у фронта нет: всё работает на нативных модулях браузера (`<script type="module">`).

## Страницы и их связи

Фронт состоит из четырёх HTML-страниц — это четыре независимых «экрана», между которыми пользователь переходит по обычным ссылкам. Состояние сохраняется в `localStorage`, поэтому при переходе между страницами ничего не теряется.

```
index.html ──► profile.html ──► recommendations.html ──► calendar.html
   лендинг       анкета         лента олимпиад         план + дедлайны
                       └──────────────────────────────────►┘
                              (план пополняется со страницы рекомендаций)
```

### `index.html` — лендинг
Статическая страница: описание продукта и кнопка «Начать» → ведёт на `profile.html`. JS здесь минимальный, не подключает модули данных.

### `profile.html` + `js/profile.js` — анкета
- При загрузке:
  - подтягивает справочники регионов и предметов через `Api.getRegions()` / `Api.getSubjects()`;
  - если у пользователя уже есть сохранённый профиль — заполняет форму через `Api.getProfile()`.
- При сабмите:
  - собирает объект `{ name, grade, region, level, past, subjects[] }`;
  - сохраняет через `Api.saveProfile()`;
  - показывает toast «Профиль сохранён» и редиректит на `recommendations.html`.

### `recommendations.html` + `js/recommendations.js` — лента
- Грузит список олимпиад через `Api.getRecommendations(filters)`.
- Рендерит карточки: название, предметы, классы, регион, дедлайн, теги, релевантность (`score`).
- Фильтры (предмет, класс, регион, поиск) дёргают тот же эндпоинт с query-параметрами — есть `debounce` на поиске.
- Кнопка **«Почему мне?»** на карточке вызывает `Api.explain(id)` и показывает LLM-обоснование в модалке.
- Кнопка **«В план»** вызывает `Api.addToPlan(id)` → олимпиада появится в `calendar.html`.

### `calendar.html` + `js/calendar.js` — план и дедлайны
- Тянет план пользователя через `Api.getPlan()`.
- Группирует элементы по месяцам функцией `groupByMonth()` из `utils.js`.
- Для каждой карточки показывает «осталось N дней» через `deadlineLabel()`.
- Кнопка «Убрать» → `Api.removeFromPlan(id)`.
- Кнопка экспорта собирает `.ics`-файл (импорт в Google Calendar / Apple Calendar).
- При открытии страницы автоматически срабатывает `checkUpcoming()` из `notifications.js` — показывает тосты по олимпиадам с дедлайном ≤14 дней.

## Модули данных

### `js/api.js` — ядро
Единственный файл, который общается с сетью. Все остальные модули ходят к нему, а не к `fetch` напрямую.

- `Auth.getUserId()` — берёт `user_id` из `localStorage`, при первом заходе генерирует случайный (`u_xxxxxxxx`). Это упрощённая «авторизация» для прототипа: каждый браузер = свой пользователь.
- `Api.*` — объект с методами под каждый эндпоинт:
  - `getProfile()` / `saveProfile(profile)`
  - `getRecommendations(filters)` / `explain(olympiadId)`
  - `getPlan()` / `addToPlan(id)` / `removeFromPlan(id)`
  - `getNotifications()`
  - `getRegions()` / `getSubjects()`
- Внутренняя функция `request()` оборачивает `fetch`: проставляет `Content-Type: application/json`, сериализует тело, бросает осмысленную ошибку при не-2xx ответе.
- Базовый префикс — `/api`, который Nginx проксирует на бэкенд.

### Mock-режим (там же, в `api.js`)
Чтобы фронт работал без бэка, в `api.js` встроены данные-заглушки: `MOCK_OLYMPIADS` (6 олимпиад), `MOCK_REGIONS`, `MOCK_SUBJECTS`, `MOCK_REASONS` (LLM-обоснования).

Каждый метод `Api.*` обёрнут так:

1. если mock включён — сразу отдаёт mock-данные;
2. иначе пробует реальный запрос;
3. **если реальный запрос упал** — автоматически включает mock (`enableMock()`) и отдаёт mock.

Это значит: даже если бэкенд не поднят, страницы открываются и наполняются — это удобно для демо и отладки UI.

Принудительно включить / выключить:
```js
localStorage.setItem('mock', '1');   // включить
localStorage.removeItem('mock');     // выключить
```

Профиль и план в mock-режиме сохраняются в `localStorage` (`profile`, `plan`).

### `js/utils.js` — хелперы без состояния
- **Даты:** `formatDate`, `daysLeft`, `deadlineLabel` (`«сегодня» / «завтра» / «через N дн.» / полная дата`).
- **Константы месяцев** в трёх формах (родительный/именительный/сокращённый) для красивого вывода.
- **DOM:** `el(tag, opts, children)` — фабрика элементов, чтобы не плодить `document.createElement`; `$`, `$$`, `clear`.
- **`groupByMonth(items)`** — группировка для календаря.
- **`debounce(fn, ms)`** — для поиска в фильтрах.
- **`skeletonCard()`** — плейсхолдер при загрузке.

### `js/notifications.js` — оповещения
- `toast(title, body, type, ttl)` — рисует тост в правом верхнем углу (`.toasts` контейнер создаётся лениво).
- `checkUpcoming()` — запрашивает `Api.getNotifications()` и показывает до трёх ближайших дедлайнов. Тосты с `daysLeft ≤ 3` подсвечиваются как `warning`.

## Поток данных (end-to-end)

```
профиль:
  profile.html ──► profile.js ──► Api.saveProfile() ──► /api/profile
                                                   └──► localStorage (бэкап + mock)

рекомендации:
  recommendations.html ──► recommendations.js
                              ├──► Api.getRecommendations(filters) ──► /api/recommendations
                              ├──► Api.explain(id)                  ──► /api/explain
                              └──► Api.addToPlan(id)                ──► /api/plan

календарь:
  calendar.html ──► calendar.js
                       ├──► Api.getPlan()           ──► /api/plan
                       ├──► Api.removeFromPlan(id)  ──► /api/plan (DELETE)
                       └──► экспорт .ics (локально, без сети)

тосты:
  любая страница ──► notifications.js.checkUpcoming() ──► Api.getNotifications()
```

Всё проходит через `api.js`. Никакой другой модуль не делает `fetch` напрямую — это упрощает переключение на mock и единое логирование ошибок.

## Стилизация

- `css/reset.css` — нормализация дефолтных стилей браузера.
- `css/variables.css` — палитра и токены (цвета, радиусы, тени, отступы). Менять тему = править этот файл.
- `css/main.css` — компоненты: карточки, формы, кнопки, чипы, модалки, тосты, сетка календаря. Вёрстка mobile-first, на десктопе разворачивается в сетку.

## Запуск

### Через docker-compose (вместе с бэком)
```bash
docker-compose up --build
```
Открыть: <http://localhost:3000>

### Только фронт (демо без бэка)
Достаточно поднять любую статику в папке `frontend/`:
```bash
cd frontend
python -m http.server 8000
```
И открыть <http://localhost:8000>. Фронт сам уйдёт в mock-режим, как только увидит, что `/api` не отвечает.

> Открывать `index.html` двойным кликом (через `file://`) **нельзя**: браузер заблокирует ES-модули и `fetch` из-за CORS. Нужен HTTP-сервер.

### Режим разработки
В docker-compose папка `frontend/` смонтирована как volume — правки HTML/CSS/JS видны после **F5**, без пересборки контейнера.
